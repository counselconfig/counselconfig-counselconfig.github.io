5.1     Web Service Report

 

 

5.1.1  Web Service basic explanation  

 

 

Somewhat plainly, the opposite of a ‘web site’ is a ‘web service’. The former is accessed by people through a user interference (UI), and the latter is accessed by computers through an application programming interface (API). Call it a B2B conversation, so when some application asks for data from a server, and the client code processes the command, regardless of its language, a Web Service is used (Manes, 2003, pp. 34-35).  This is because the representation of communication is a layer that backs loose coupled connections. It means the client and server need limited knowledge about each other as modules to bridge a request for data. (Jalote, 2008, p. 123). The way applications talk in a Web Service is therefore simplified, which will result in lower management costs and reduced savings for Wallace’s Widgets Company float.  

 

 

5.1.2  Web Service evaluation

 

For a vehicle hire Web Service to work, we need processes that can integrate ERP applications so vehicle pool data can accessed concurrently by customers through a conduit that is the HTTP.

First, the Web Service requires scalability to leverage performance. To cope with high user traffic a connection between several servers can distribute the store and process load horizontally. Given the cheapness of peer databases, this might be attractive to the company float, but we are forewarned not to overkill with sharding (Lake & Crowther, 2013, p. 216). Furthermore, a balancer could aid requests for service across a number of servers to ‘improve’ distribution by splitting process load equally. An OSI model Layer 2 or 4 load balancer could offset high volume transactions to diminish response depreciation (Bourke, 2001, pp. 13-23).

The third problem is that HTTP only describes information for display without meaning. As I see it, this leaves a void where the Web Service cannot get data without being guided as to where it should go and what information to takeaway. That is why the W3C (2004) standardised three protocols in XML to make this possible, viz. Simple Object Access Protocol (SOAP), Web Services Description Language (WSDL), and Universal Description, Discovery & Integration directory (UDDI): Turner, et al. (2003).

SOAP

 

When the SOAP is actioned, vehicle hire data is interchanged in an XML over the HTTP that communicates input/output messages. The XML will mark this data contained in self-descriptive meta-labels, readable by humans and machines. This way, the Web Service can exchange data with applications using different programming languages to manipulate and import into their ERP’s with some sense.

 

<!--  ERP company request message for a car on a certain date -->

<?xml version="1.0"?>

<soap:Envelope
xmlns:soap="http://www.w3.org/2003/05/soap-envelope/"
soap:encodingStyle="http://www.w3.org/2003/05/soap-encoding">

<soap:Body>
  <m:soapGetVehicles xmlns:m="https://www.wallacesvehiclehire.com/Vehicles">
    <m:BookingFor_Date>2018/03/19</m:BookingFor_Date>
  </m:soapGetVehicles>
</soap:Body>

</soap:Envelope>

However, lastly, should errors occur with the call request, then the situation may need debugging. Unlike the REpresentational State Transfer, SOAP includes built in exception handling. This can be achieved through Fault elements that describe exceptional predicaments nested in the message that allows developers to diagnose exceptions (Erl , 2004, pp. 78-80).  Example, if the Vehicle hire service experiences a fault with the server – technological failure related to hard disks are common (Lake & Crowther, 2013, p. 255) - the SOAP gives a human readable often a generic faultstring output:

 

 

<?xml version="1.0" encoding="UTF-8"?>

<soap:Envelope

 xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/" >

  <soap:Body>

    <soap:Fault>

     <faultcode>soap:Server</faultcode>

     <faultstring>Wallaces Widget’s database is unavailable?</faultstring>

   </detail>

  </soap:Fault>

 </soap:Body>

</soap:Envelope>

 

Figure 5.10 Server internal error

 

In HTTP terms, this triggers a ‘500 Internal Server Error’ message received by the customer (Mueller, 2002, p. 38).  The other kind of error is that of ACID transactions at the data tier, since many users may make a GET request for vehicles on the same date concurrently. In such circumstances, in terms of ‘A’, for Atomicity, a transaction ends with a COMMIT; or a ROLLBACK; specified to cancel an operation on a table where an error occurs. Usually remedied by shared and exclusive ‘locks’, this might be where two transactions (T) may GET (G), then POST (P), one booking each for a vehicle having read there are five left on 2018/03/19:
